# Skill 最適化フレームワーク

> Claude Code の Skill（SKILL.md）を体系的にイテレーション改善する手法。
> design-review Skill の最適化実績（82→98, 4イテレーション）を汎用化したもの。

---

## 概要

**手法**: ProTeGi/APO（Pryzant et al., EMNLP 2023）のテキスト勾配アプローチを Skill プロンプト最適化に応用する。

**核心ループ**:
```
SKILL.md（ベースライン）
  → テストケースで実行
    → expected と比較・採点
      → テキスト勾配（根本原因分析 + 修正提案）
        → SKILL.md を修正 → 再実行
          → 収束するまで繰り返す
```

**典型的な収束**: 3〜4 イテレーションでスコア 90+ に到達する。

---

## Phase 1: 準備

### 1-1. ディレクトリ構造を作る

```
.claude/skills/{skill-name}/
├── SKILL.md                    # 本体（Claude が読むプロンプト）
├── tasks/
│   ├── evaluate.md             # 評価基準（軸・重み・収束条件）
│   ├── case_01/
│   │   ├── input.*             # 入力データ
│   │   └── expected.md         # 期待される出力
│   ├── case_02/
│   │   ├── input.*
│   │   └── expected.md
│   └── case_03/
│       ├── input.*
│       └── expected.md
└── iterations/
    ├── iter_0/
    │   ├── skill.md            # このイテレーションの SKILL.md スナップショット
    │   └── gradient.md         # テキスト勾配（分析 + 改善提案）
    ├── iter_1/
    │   ├── skill.md
    │   └── gradient.md
    └── ...
```

### 1-2. テストケースを設計する（3件セット）

| ケース | 役割 | 設計指針 |
|--------|------|----------|
| **case_01** | 網羅性テスト | 多数の入力パターン・エッジケースを含む複雑な入力。Skill が対応すべき全カテゴリをカバーする |
| **case_02** | 精度テスト | ほぼ正しい入力（違反・問題が少数）。過検出（偽陽性）しないかを検証する |
| **case_03** | 境界テスト | 微妙な判断が必要な入力。グレーゾーンでの Skill の判断力を検証する |

**設計のコツ**:
- case_01 は 10〜20 項目の検出対象を含む（少なすぎると completeness の差が出にくい）
- case_02 は「正しい実装」を多く含み、2〜5 項目だけ問題がある状態にする
- case_03 は文脈依存の判断（同じパターンでも OK/NG が分かれるもの）を含める
- 各 `expected.md` には検出すべき項目、重大度、修正方法を明記する

### 1-3. 評価基準を定義する（evaluate.md）

**テンプレート**:

```markdown
# 評価基準

## 評価軸（各0〜100点、加重平均でスコア算出）

| 軸 | 重み | 見るポイント |
|----|------|------------|
| {軸1} | {重み}% | {何を測定するか} |
| {軸2} | {重み}% | {何を測定するか} |
| ... | ... | ... |

## スコアリング手順

1. Skill の出力と expected.md を並べる
2. 各軸を 0〜100 で採点する
3. 加重平均で総合スコアを算出する
4. テキスト勾配として差分の根本原因と改善提案を記述する

## 収束条件

- 総合スコア {目標} 以上 かつ 全ケースで {主要軸} {閾値} 以上
- または 2イテレーション連続でスコア改善が 1 点未満（収穫逓減）
```

**軸の選び方ガイド**:

| Skill の種類 | 推奨軸（重み順） |
|-------------|-----------------|
| レビュー・検証系 | completeness(25%), accuracy(25%), severity(15%), actionability(15%), structure(10%), conciseness(10%) |
| 生成系 | correctness(30%), completeness(25%), style_adherence(20%), structure(15%), conciseness(10%) |
| 変換・リファクタ系 | correctness(30%), completeness(25%), preservation(20%), readability(15%), conciseness(10%) |
| 分析・要約系 | accuracy(30%), completeness(25%), relevance(20%), structure(15%), conciseness(10%) |

> 重みの合計は 100% にすること。タスク固有の軸を自由に追加・変更してよい。

## Phase 2: ベースライン作成（iter_0）

### 2-1. SKILL.md の初版を書く

以下の構造を推奨:

```markdown
---
name: {skill-name}
description: {1行の説明}
---

# {Skill名}

{Skill の目的を1〜2文で}

## 手順

### Step 1: 入力を読む
{入力データの読み込み方法}

### Step 2: 参照情報を読む
{判断基準となる参照データの読み込み}

### Step 3: 処理を実行する
{メインのロジック — チェックリスト、生成ルール等}

### Step 4: 結果を検証する
{偽陽性防止、品質チェック等}

### Step 5: 出力する
{出力フォーマットの指定}
```

**初版では完璧を目指さない**。基本的な構造とルールだけ書き、不足点はイテレーションで補う。

### 2-2. 全テストケースで実行し、gradient を生成する

```
1. case_01/input.* を SKILL.md で処理 → 出力を保存
2. case_02/input.* を SKILL.md で処理 → 出力を保存
3. case_03/input.* を SKILL.md で処理 → 出力を保存
4. 各出力を expected.md と比較し gradient.md を生成
```

## Phase 3: イテレーション

### 3-1. テキスト勾配のフォーマット（gradient.md）

```markdown
## テキスト勾配（イテレーション N）

### スコア
| 軸 | Case 01 | Case 02 | Case 03 | 平均 |
|----|---------|---------|---------|------|
| {軸1} | XX | XX | XX | XX |
| ... | ... | ... | ... | ... |
| **総合** | XX | XX | XX | **XX** |

### 検出漏れ / 不足（False Negatives）
- [具体的にどの項目を見逃したか、なぜ見逃したと推定されるか]

### 誤検出 / 過剰（False Positives）
- [正しいものを誤って問題と報告した箇所、なぜ誤報したと推定されるか]

### 構造・表現の問題
- [出力の構造や表現で改善すべき点]

### Skill改善の提案
**P1（即時対応）** — {主要軸}に直結
1. [具体的な修正指示]
2. [具体的な修正指示]

**P2（次イテレーション）** — {副次軸}に直結
3. [具体的な修正指示]

**P3（将来対応）** — 収束後の品質維持
4. [具体的な修正指示]

### 収束判定
| 条件 | 状態 |
|------|------|
| 総合スコア {目標} 以上 | 達成 / 未達 |
| 全ケース {主要軸} {閾値} 以上 | 達成 / 未達 |
| 2イテ連続で改善 1pt未満 | 該当 / 非該当 |

**判定: 収束 / 未収束。次イテレーションへ進む。**
```

### 3-2. 改善の優先順位付け

| 優先度 | 基準 | 改善タイミング |
|--------|------|--------------|
| **P1** | 主要評価軸（重み25%以上）のスコアに直結 | 当該イテレーションで即時適用 |
| **P2** | 副次評価軸（重み15%程度）のスコアに直結 | 次イテレーションで適用 |
| **P3** | 品質向上だがスコアへの影響が小さい | 収束後に任意で適用 |

### 3-3. SKILL.md の修正ルール

- **P1 の改善だけを適用** してから再実行する（一度に多くの変更を入れると原因特定が困難になる）
- 各イテレーションの SKILL.md スナップショットを `iterations/iter_N/skill.md` に保存する
- 前イテレーションの gradient.md で指示された P1 が反映されたことを確認してから実行する

### 3-4. 収束パターン（design-review での実績）

```
iter_0: 82（ベースライン — 構造的な不足が多い）
iter_1: 88（+6 — チェックリスト拡充・出力構造化）
iter_2: 94（+6 — 偽陽性防止・重大度修正）
iter_3: 98（+4 — エッジケース対応・最終調整）→ 収束
```

**典型的なフェーズ遷移**:
| フェーズ | イテレーション | 改善内容 | 期待される改善幅 |
|---------|--------------|---------|----------------|
| 構造改善 | iter_0→1 | チェックリスト追加、出力フォーマット定義、基本ルール整備 | +5〜10 |
| 精度向上 | iter_1→2 | 偽陽性フィルタ、文脈判定、ルール微調整 | +4〜8 |
| 最終調整 | iter_2→3 | エッジケース対応、ホワイトリスト、固定ルール化 | +2〜5 |
| 収束 | iter_3→ | 改善幅 < 2 で停止 | — |

## Phase 4: 収束と完了

### 4-1. 収束条件の判定

以下の **いずれか** を満たしたら完了:
1. 総合スコアが目標以上 かつ 全ケースで主要軸が閾値以上
2. 2イテレーション連続でスコア改善が 1 点未満

### 4-2. 完了時の作業

1. 最終版 SKILL.md を `.claude/skills/{skill-name}/SKILL.md` に配置
2. `iterations/` にすべての履歴が保存されていることを確認
3. MEMORY.md にスコア推移・主要な学びを記録

## 適用可能性ガイド

### 高い再現性が期待できるタスク

| タスク種別 | 理由 |
|-----------|------|
| ルールベースのレビュー・検証 | 「正解」が明確に定義できる（規約、仕様書、禁止パターン等） |
| コード生成（テンプレート的） | 出力の構造が予測可能で、正誤判定が容易 |
| テスト生成 | カバレッジ・パス数等で定量評価可能 |
| データ変換 | 入出力の対応関係が明確 |

### 再現性が限定的なタスク

| タスク種別 | 困難な点 | 対処法 |
|-----------|---------|--------|
| 自由記述の生成（文章・デザイン） | 「正解」が主観的 | 構造・形式の軸は評価可能。内容の質は人間レビューを挟む |
| 対話・質問応答 | 文脈依存が強い | テストケースを会話形式にし、「含むべき情報」で評価する |
| 探索的タスク（調査・分析） | 出力が非決定的 | 「最低限含むべき発見」をexpectedに定義する |

### 「期待出力」を定義するコツ

1. **完全一致ではなく「含むべき要素」で定義する** — 文言の一致ではなく、検出すべき項目リストで評価する
2. **重大度・カテゴリを明示する** — 内容だけでなく分類も期待出力に含める
3. **「報告しないべき項目」も書く** — 偽陽性の評価に必要
4. **曖昧な項目には判断根拠を付記する** — Skill と evaluator の解釈ブレを防ぐ

## 参考: design-review での具体的な学び

### 効果的だった改善パターン

1. **チェックリストの明示化** — 暗黙のルールを箇条書きにすると completeness が大幅向上（+7〜10）
2. **偽陽性防止ステップの追加**（Step 4-2）— 「報告前に文脈を確認する」指示で accuracy が +4
3. **重大度の固定ルール化** — 「この違反は必ず X にする」と例外なしで指定すると severity が +23
4. **出力フォーマットの厳格化** — テンプレートを提示すると structure が +20
5. **集約ルール** — 「同一違反は1件に集約」で conciseness が +11

### 陥りやすい落とし穴

1. **偽陽性抑制の過剰適用** — iter_1 で「文脈判定して除外」を強化しすぎて、iter_2 で本来検出すべき項目まで除外してしまった（nav aria-label の例）
2. **箇所数カウントの定義曖昧** — 「何をカウント対象とするか」を明記しないと、イテレーションごとに数値がブレる
3. **暗黙の severity 推論** — Skill が独自に重大度を推定すると不安定。固定テーブルで上書きする方が確実
