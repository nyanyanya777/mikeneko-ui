### 検出漏れ（False Negatives）

**Case 01 — 3件の漏れ**

1. **`p-4` on cards（スペーシング）**
   - expected.md の項目 12「`p-4` on cards — コンテンツが窮屈（p-5未満）→ `p-5` 以上」が未検出。
   - 推定原因: `rounded-none` と `shadow-lg` の検出に注力する一方で、パディング値のチェックルールがないか優先度が低い。カードの `p-4` は許容値に見えるが DS では `p-5` 以上が基準であることを Skill が把握していない可能性がある。

2. **`<nav>` に `aria-label` なし（アクセシビリティ）**
   - expected.md の accessibility セクションに明記されているが未検出。
   - 推定原因: aria-current は検出できているため `<nav>` 要素自体は認識している。`aria-label` の有無まで検査するルールが Skill に含まれていない。

3. **`<th>` に `scope` なし（アクセシビリティ）**
   - expected.md の accessibility セクションに明記されているが未検出。
   - 推定原因: テーブルのアクセシビリティ検査ルールが不完全。`<th>` に対する `scope="col"` / `scope="row"` の検証が Skill のチェックリストに存在しない。

**Case 02 — 0件**

全 4 件を正しく検出。

全 4 件を正しく検出。

### 誤検出（False Positives）

**Case 02 — 2件の誤検出**

1. **チェックボックスサイズ `w-4 h-4` → CSS 18px（Case 02 出力 項目 5）**
   - expected.md はこの違反を含まず、「良い実装」セクションでも言及なし。
   - 推定原因: Skill が `w-4 h-4` をチェックボックスの不正サイズと判断したが、Case 02 の HTML では CSS で 18px が設定されている場合など文脈を正確に読み取れなかった可能性がある。Tailwind クラスだけでなく `<style>` ブロックや外部 CSS のオーバーライドを考慮しない静的解析の限界。

2. **ボタンの `inline-flex items-center justify-center gap-2` 欠落（Case 02 出力 項目 6）**
   - expected.md に含まれない違反。expected.md の「良い実装」にボタンのパディングが正しいとの記載あり。
   - 推定原因: Skill がボタンの必須クラスを機械的にチェックする際、`inline-flex` 系クラスを構造的必須要件とみなした。しかし DSのクイックリファレンスは推奨パターンであり、アイコンなしのテキストのみボタンでは必ずしも必須ではない。Skill が「推奨」と「必須」の区別を付けられていない。

**Case 01 — 0件（ただし構造的重複あり）**

誤検出はないが、`border-gray-100` を項目 13（テーブル行）と項目 14（テーブルラッパー）の 2 件に分割報告している。これは同一の違反カテゴリを場所別に列挙したもので、conciseness に影響する（後述）。

**Case 03 — 0件**

誤検出なし。

### 構造・表現の問題

1. **重大度分類の精度が低い（全ケース共通、severity 平均 60）**
   - Case 01 で期待値は Critical 3 / High 7 / Medium 5 / Low 1 に対し、Skill は Critical 4 / High 9 / Medium 4 を報告。Low カテゴリが消滅し、Critical と High が水増しされている。
   - `border-gray-100` は視覚的影響が小さい Low 相当であるのに High として分類されたと推定される。重大度判定のルールが曖昧で、「視覚的影響」「アクセシビリティ影響」「UX影響」の3軸に基づく明確な判定基準が Skill に欠けている。

2. **カテゴリ別の構造化がない（全ケース共通、structure 平均 60）**
   - 全ケースで出力が「フラットな番号付きリスト」のみ。expected.md は「カラー」「サイドバー」「スペーシング・レイアウト」「ボタン」「ボーダー」「フォーム」「アクセシビリティ」のカテゴリに整理されている。
   - 行番号の参照がない。HTMLのどの行に違反があるかを示していない。
   - サマリー（カテゴリ別件数テーブル）がない。

3. **同一違反の複数箇所を列挙するか集約するかの一貫性がない（conciseness）**
   - Case 01 で `border-gray-100` を 2 件に分けて報告しつつ、`text-black` の 6 箇所、`text-gray-400` の 5 箇所は 1 件にまとめている。集約ルールが不統一。
   - 推奨: 同一違反カテゴリは 1 件にまとめ、括弧内に「（X箇所）」と件数を示す形式に統一する。

4. **フォームの `<label>` / `focus:outline-none` / `appearance-none` を 1 項目にまとめすぎ（Case 01 項目 17）**
   - 3 つの独立した違反（label 欠如、focus ring 消失、appearance-none 欠如）が 1 項目に混在している。各違反は重大度が異なる（label 欠如 = Critical、focus ring = Critical、appearance-none = High）ため、分離して報告すべき。

### Skill改善の提案

以下を優先度順に Skill のプロンプトまたはチェックリストへ反映することを推奨する。

**P1（即時対応）— completeness と accuracy に直結**

1. **アクセシビリティチェックリストの拡充**
   - 現行: `aria-current`, `<label>`, `focus:ring` を検出済み
   - 追加: `<nav aria-label="">` の存在確認、`<th scope="col|row">` の確認、`role="group"` または `<fieldset>/<legend>` の確認を独立したチェック項目として追加する。

2. **「推奨」と「必須」の区別ルール**
   - ボタンの `inline-flex items-center justify-center gap-2` はアイコン付きボタンの推奨パターンであり、テキストのみボタンには必須でない旨を Skill に明示する。
   - チェック条件: ボタン内に `<svg>` またはアイコン要素がある場合のみ `gap-2` / `items-center` を検査する。

3. **`p-4` on cards の検出ルール追加**
   - カードコンポーネントのパディングが `p-5` 未満（`p-0`, `p-2`, `p-3`, `p-4`）の場合を違反とするルールを追加する。

**P2（次イテレーション）— severity と structure に直結**

4. **重大度判定の明確化**
   - 重大度を以下の基準で判定するよう Skill に指示する:
     - Critical: スクリーンリーダー・キーボード操作に直接影響（label 欠如、focus 消失、aria 欠如）
     - High: DS カラー体系またはセマンティクスの違反（text-black, text-gray-400, 色名不統一、暗背景）
     - Medium: スペーシング・エレベーション・サイズの不統一（rounded-none, shadow-lg, p-4, py-0.5, primary-400）
     - Low: 境界線・細部の視覚的なずれ（border-gray-100, 非標準幅）
   - 上記をプロンプト内に severity lookup table として埋め込む。

5. **出力フォーマットの構造化**
   - 以下のフォーマットを Skill の出力仕様として指定する:
     ```
     ## レビュー結果サマリー
     | 重大度 | 件数 |
     |--------|------|
     | Critical | N |
     | High | N |
     | Medium | N |
     | Low | N |

     ## 違反詳細

     ### カラー
     - [重大度] **`違反クラス`**（N箇所）— 理由 → `修正案`
       - 行: XXX

     ### アクセシビリティ
     ...
     ```

6. **同一違反の集約ルール**
   - 同一の違反クラス（例: `border-gray-100`）が複数箇所にある場合は 1 件として集約し、「（N箇所: 行 XX, XX, XX）」と付記する。
   - 同一違反を複数項目に分割して報告しない。

**P3（将来対応）— actionability の向上**

7. **修正コードスニペットの標準化**
   - 現状では「→ `text-slate-900`」のようなインライン修正のみ。
   - `appearance-none` のような複合修正（ラッパー div + SVG シェブロン追加が必要）については、複数行のコードブロックを提示するよう Skill に指示する。
   - 例: select の appearance-none 修正は `<div class="relative">` + `<select class="appearance-none pr-10">` + `<svg>` のセット提示を必須とする。
